# Dr.Slon Toolkit

Dr.Slon Toolkit — модульный плагин WordPress для обслуживания и базового усиления клиентских сайтов.

## Текущая версия (0.9.0)

Первая рабочая версия включает:

- минимальную точку запуска с автозагрузкой Composer (PSR-4)
- центральный класс запуска плагина
- обработчики активации и удаления
- нативный адаптивный интерфейс и отдельную страницу помощи
- переключатели модулей через Settings API
- рабочие безопасные модули:
  - Скрытый вход
  - REST API Control
  - IndexNow
  - Sitemap
  - Update Controls
  - Транслитерация
  - Отключение комментариев
  - Очистка
- интеграцию с The SEO Framework для Sitemap и IndexNow
- проверяемые обновления из GitHub Releases
- локальные карточки поддержки без удалённого JavaScript

## Требования

- WordPress 6.6+
- PHP 8.1+

## Установка у клиента

Composer на сервере клиента **не нужен**.
Клиенту передаётся готовый ZIP-архив плагина, который устанавливается через стандартный установщик WordPress.

## Что уже реализовано

### Скрытый вход
- Возвращает 404 для прямого доступа к `wp-login.php`, кроме необходимых WordPress reset/recovery действий.
- Открывает страницу входа по настраиваемому slug, например `/my-login/`.
- Сохраняет рабочие сценарии logout и восстановления пароля.
- Имеет аварийное отключение через константу `KRV_DSTK_DISABLE_HIDE_LOGIN` в `wp-config.php`.
- При смене slug автоматически обновляет rewrite rules.

#### Ручная проверка «Скрытого входа»
1. Включите модуль и задайте slug, например `my-login`, затем сохраните настройки.
2. Проверьте, что `/my-login/` открывает форму, а прямой `wp-login.php` возвращает 404.
3. Проверьте вход, выход, `lost password` и переход по ссылке `reset password`.
4. Проверьте доступ администратора в `/wp-admin/` после входа.
5. Для аварийного доступа добавьте в `wp-config.php`:
   `define('KRV_DSTK_DISABLE_HIDE_LOGIN', true);`

### REST API Control
- Добавляет управляемый доступ к REST API без полного отключения API.
- Поддерживает режимы:
  - разрешить всем;
  - только авторизованным;
  - ограниченный режим по whitelist.
- Позволяет задать:
  - whitelist точных маршрутов;
  - whitelist namespace;
  - доверенную capability для обхода ограничений;
  - дополнительные безопасные маршруты поверх встроенного системного allowlist.
- При запрете отдаёт корректный REST-ответ с ошибкой доступа.
- Встроенный системный allowlist WordPress всегда активен и не зависит от пользовательского поля.

#### Ручная проверка «REST API Control»
1. Включите модуль и выберите режим «Только авторизованным».
2. Проверьте REST-запрос гостем к `/wp-json/wp/v2/posts` — должен вернуться отказ доступа.
3. Проверьте тот же запрос авторизованным пользователем — должен работать.
4. Переключите на режим whitelist и добавьте маршрут `/wp/v2/posts`.
5. Проверьте, что разрешены только whitelist/system-маршруты.
6. Убедитесь, что редактор (Gutenberg) работает у пользователя с capability из настройки.

### IndexNow
- Поддерживает отправку URL в IndexNow через WordPress HTTP API.
- Работает с ключом IndexNow и endpoint (универсальный, Bing, Yandex).
- Автоматически отправляет URL при публикации/обновлении/удалении поддерживаемых типов записей.
- Автоматические запросы выполняются через очередь WP-Cron и не задерживают редактор.
- Позволяет вручную отправить один URL из админки.
- Публикует проверочный ключ по URL вида `/<key>.txt` без записи файлов на диск.
- Использует встроенный whitelist endpoint и защиту от частых повторных отправок одного и того же URL.
- При активном The SEO Framework пропускает noindex и URL с отличающимся canonical.

### Sitemap
- Добавляет XML sitemap index по адресу `/sitemap.xml`.
- Формирует отдельные sitemap для включённых сущностей:
  - типы записей: `/sitemap-pt-{post_type}.xml`
  - таксономии: `/sitemap-tax-{taxonomy}.xml`
- Позволяет в настройках выбрать типы записей и таксономии для включения.
- Разбивает большие наборы на страницы по 1000 URL и кеширует XML-ответы.
- Исключает записи со статусом, отличным от `publish`, и записи с паролем.
- Для исключения noindex предусмотрен фильтр `dstk_sitemap_is_noindex`.
- Добавляет `lastmod` для URL записей и для sitemap index, где дата доступна безопасно.
- При активном The SEO Framework работает в safe mode: runtime sitemap Dr.Slon Toolkit не отдаётся, чтобы избежать дублей.

### Update Controls
- Управляет автообновлениями через нативные фильтры WordPress.
- Поддерживает режимы для ядра: все, minor, security/критические (без major/dev), полностью отключить.
- Позволяет запрещать автообновления плагинов, тем и переводов, сохраняя защитные решения WordPress при разрешении.
- Позволяет включать/выключать e-mail уведомления об автообновлениях.
- По умолчанию использует консервативные безопасные значения (minor для ядра, остальные автообновления и уведомления включены).

### Транслитерация
- Использует практический русский URL-профиль (`Главная` → `glavnaya`).
- Преобразует нелатинские slug терминов при необходимости.
- Преобразует имена загружаемых файлов в безопасный латинский вид.
- Подключается к формированию slug при сохранении, чтобы корректно обрабатывать сложные случаи с кириллическими заголовками.
- Не изменяет уже опубликованные URL автоматически.

### Отключение комментариев
- Глобально закрывает комментарии и пинги.
- Убирает поддержку комментариев и трекбеков у типов записей.
- Скрывает меню комментариев и пункт комментариев в верхней панели.
- Перенаправляет экраны управления комментариями в консоль.

### Очистка
- Даёт отдельные переключатели для:
  - отключения emoji
  - отключения скрипта wp-embed
  - отключения XML-RPC
  - удаления выбранных безопасных тегов из `<head>`

## Разработка

```bash
composer install
composer check
```

После этого активируйте плагин и настройте модули в меню **Dr.Slon Toolkit** в wp-admin.

## Сборка релизного ZIP (для клиента)

Composer используется только на этапе разработки и сборки.

Вариант 1 (через Composer-скрипт):

```bash
composer build-release
```

Вариант 2 (напрямую):

```bash
bash tools/build-release.sh
```

Сценарий сборки:
- удаляет предыдущую сборку в `build/` и старый архив в `dist/`;
- ставит production-зависимости (`--no-dev`) только внутри staging-каталога;
- формирует самодостаточный архив `dist/dr-slon-toolkit-<version>.zip`;
- упаковывает файлы внутри корневой папки `dr-slon-toolkit/`.
- проверяет согласованность версии в заголовке плагина, `DSTK_VERSION` и `Stable tag`, безопасную структуру архива, его размер, assets и runtime-autoload.

## Обновления

Начиная с `0.9.0`, WordPress получает сведения о новых версиях из GitHub Releases. Устанавливается только asset с точным именем `dr-slon-toolkit-<version>.zip`; автоматически созданные GitHub source archives не используются, потому что в них отсутствует production `vendor/`.

SHA-256 из GitHub API используется как контрольная сумма загруженного asset, а не как криптографическая подпись издателя. Релиз может оставаться изменяемым: установщик принимает только пакет, совпадающий с метаданными конкретного сохранённого предложения обновления.

Переход с `0.8.2` на первый updater-enabled релиз выполняется вручную. Последующие версии появляются в стандартном интерфейсе обновлений WordPress.

В архив включаются:
- `dr-slon-toolkit.php`
- `readme.txt`
- `uninstall.php`
- `src/`
- `vendor/` (включая `vendor/autoload.php`)
- `languages/` (если есть)
- `assets/` (если есть)
- `LICENSE`

Из архива исключаются:
- `.git/`, `.github/`
- `tests/`
- `AGENTS.md`
- `build/`, `dist/`
- служебные файлы редакторов, логи и прочие временные файлы
- документация разработки, не нужная для работы плагина
